三、git基本知识

使用远程仓库时候会有多个协议可以选择，使用https不仅仅速度慢，而且每次push都要输入口令。

3.1、git关键字解释

HEAD 当前版本的指针，当切换本地版本的时候会快速指向指定版本文件

master git为我们创建主分支

origin 远程仓库的名称

3.2、git文件的四种状态

  • Untracked：未跟踪, 此文件在文件夹中, 但并没有加入到git库, 不参与版本控制. 通过git add 状态变为Staged.
  • Unmodify：文件已经入库, 未修改, 即版本库中的文件快照内容与文件夹中完全一致. 这种类型的文件有两种去处, 如果它被修改, 而变为Modified. 如果使用git rm移出版本库, 则成为Untracked文件
  • Modified：文件已修改, 仅仅是修改, 并没有进行其他的操作. 这个文件也有两个去处, 通过git add可进入暂存staged状态, 使用git checkout 则丢弃修改过, 返回到unmodify状态, 这个git checkout即从库中取出文件, 覆盖当前修改
  • Staged：暂存状态. 执行git commit则将修改同步到库中, 这时库中的文件和本地文件又变为一致, 文件为Unmodify状态. 执行git reset HEAD filename取消暂存, 文件状态为Modified

四、git常用命令

git常见操作流程如下图所示：

4.1、init && clone

`git init` 在当前目录新建一个仓库

`git init -b main`创建仓库的同时创建main分支

`git init [project-name]` 在一个目录下新建本地仓库

`git clone [url]` 克隆一个远程仓库

在git init命令之后，根目录下会自动创建一个名为.git的隐藏目录，代表本地仓库已经创建完毕~

4.2、diff

git diff命令用于查看文件哪些地方做出修改，并将修改部分用不同颜色的字体进行标记：

  • 红字：表示删除该行
  • 绿字：表示添加该行
`git diff 文件名`									# 查看该文件做了哪些修改

`git diff HEAD -- .` 							  # 查看最新本地版本库和工作区所有文件的区别

`git diff HEAD -- [file-name]` 					  # 查看最新本地版本库和工作区文件的却别

`git diff HEAD^ -- [file-name]` 				  # 查看本地上一个版本和工作区文件的却别

`git diff [local branch] origin/[remote branch]`  # 比较本地分支和远程分支的区别

4.3、status

git status用于查看当前git操作状态：

`git status [file-name]` 		# 查看指定文件状态

`git status` 					# 查看所有文件状态

4.4、add

git add命令用于将本地工作区文件添加到暂存区：

`git add [file-name1] [file-name2] ...` # 从工作区添加指定文件到暂存区

`git add .` 							# 将工作区的被修改的文件和新增的文件提交到暂存区，不包括被删除的文件

`git add -u .` 							# u指update，将工作区的被修改的文件和被删除的文件提交到暂存区，不包括新增的文件

`git add -A .` 							# A指all，将工作区被修改、被删除、新增的文件都提交到暂存区

4.5、commit

在提交commit后发现还有部分文件修改忘记提交了可以是用该命令加-m是指直接在后面写上版本的注释，不加-m的话会用一个vim打开文件让你写入massage，有未追踪的文件将会失败，需要add加入暂存区。

`git commit -m [massage]` 								# 将暂存区所有文件添加到本地仓库

`git commit [file-name-1] [file-name-2] -m [massage]` 	# 将暂存区指定文件添加到本地仓库

`git commit -am [massage]` 								# 将工作区的内容直接加入本地仓库

提交修正命令：

`git commit --amend` # 快速将当前文件修改合并到最新的commit，不会产生新的commit。

# 内容一：对提交的错误文件进行修正
`git commit --amend -m "上一次的提交信息"	`# 避免产生冗余的提交日志

# 内容二：提交时遗漏文件
`git add 遗漏文件名`
`git commit --amend -m "上一次的提交信息"`

# 内容三：提交信息错误的修正
`git commit --amend -m "无错误的提交信息"`

# 内容四：长开发周期中小提交问题
`git commit --amend -m "最新的提交信息"`	# 会覆盖原始提交信息，保持日志整洁

4.6、clean

git clean命令用于删除工作目录中没有被 tracked 的文件

`git clean -n`				# 不会删除，只是Notification，提醒

`git clean --force`
`git clean -f`				# 强制删除,删除所有未跟踪的文件

`git clean -f <路径>`		   # 可以添加指定路径

`git clean -d`				# 包含目录

在使用时，通常以下两个命令一起使用，使得工作目录和缓存区复原，回归干净的目录结构：

`git reset --hard`
`git clean -df`

4.7、log

git log命令用于查看历史提交记录，通过上下方向键滚动查看日志，通过q键退出后：

`git log` 												# 显示所有commit日志

`git log --oneline`
`git log --pretty=oneline` 								# 将日志缩写为单行显示

`git log --graph`										# 将日志图形化显示

`git log --graph --pretty=oneline --abbrev-commit` 		# 查看分支合并情况

`git log --stat`										# 输出对文件更改内容的统计

`git log --patch` | `git log -p`						# 输出文件修改的内容

`git log --oneline --decorate --graph --all` 			# 查看分叉历史，包括：提交历史、各个分支的指向以及项目的分支分叉情况。

`git log -3` 											# 查看最新3条commit日志数据

对日志进行筛选操作：

# 按照日期进行筛选
`git log --after="2023-11-11"`			
`git log --before="2023-11-11"`			
`git log --before="2023-11-11" --after="2023-11-09" `		# 组合使用

`git log --after="yesterday"`

# 按贡献者进行筛选
`git log --author='shuyan'`
`git log --author='email'`		# 这个邮箱地址可以使用不完全匹配 --author='@qq.com'

# 按提交信息进行筛选
`git log --grep='提交信息'`

4.8、reflog

git reflog ，引用日志命令，可以显示操作本地版本库的命令，包括commit和reset等，在回退版本以后又后悔找不到commit id了可以使用此命令查看历史。

简明来说，reflog记录了所有的更改（HEAD的变化以及commit的提交），但只保存在本地仓库，不能push，默认保存时间为90天！

4.9、push

push命令用于将本地工作区代码提交到远程仓库：

`git push` 									# 将文件添加到远程仓库

`git push -f` 								# 强制提交，当我们本地reset到旧的版本时，然后普通push会被拦截，因为此是本地HEAD指向比远程库还要旧

`git push origin main`						# 将本地分支main的代码提交到远程仓库，若没有该分支则创建

`git push -u origin dev` 					# -u表示将本地分支代码上传并合并到远程分支

`git push origin local_branch:remote_branch` # 推送当前本地分支到指定远程分支

`git push origin --all`						# 推送所有分支到远程仓库

在对远程分支进行管理时，如删除远程分支，会导致本地库中缓存的远程分支与实际远程分支不一致问题，因此需要同步远程分支：

`git remote prune origin`

# 也可以直接在客户端删除远程仓库分支
`git push origin -d dev`			# 删除远程仓库的dev分支

在推送内容时，项目的标签不会被自动推送到远程仓库，因此可以：

`git push origin tag_name1 tag_name2`	# 推送指定标签

`git push origin --tags`				# 推送所有标签

4.10、rm

git rm命令用于删除暂存区的文件：

`git rm --cached [file-name]` 删除暂存区的文件

`git rm -rf .` 不但删除所有暂存区的文件，还删除所有工作区的物理文件

4.11、checkout

从版本库中恢复文件：

  • 已删除（或修改），未add
  • 已删除（或修改），已add，未commit
  • 已删除（或修改），已commit
git checkout -- [file-name]
  • 最好加--，没有的话就把它当作切换分支看待，切换到另一个分支了，如果没有这个分支也会把它当作文件执行。
  • 用暂存区的文件覆盖掉工作区的文件
  • 如果暂存区没有可更新的就会用commit的文件更新工作区的文件

git checkout [branch] 切换分支

git checkout -b [new-branch-name] 创建并切换分支

git的版本库如下图所示：

第一种情况：

# 假设删除shuyan.txt
git checkout shuyan.txt

第二种情况：

# 这种情需要从仓库中寻找文件
git checkout "commit提交的id" shuyan.txt

第三种情况：

git checkout "commit提交的id" shuyan.txt

4.12、reset

当对整个版本进行操作

git reset --{soft|(mixed)|hard} HEAD
  • --soft 其中可选参数soft表示单纯的切换HEAD指向的commit-id
  • --mixed 默认值mixed参数表示先执行上面一步，然后再将commit-id里面的内容更新到暂存区
  • --hard hard表示先执行上面两步，然后再将暂存区内容同步到工作区

git reset --hard HEAD^^ 用上两个版本里的所有文件撤回到暂工作区

git reset --hard [commit id] 用指定版本的所有文件撤回到工作区

当对当个文件进行操作

git reset HEAD [file-name]
  • 对单个文件操作时候只能用mixed参数，而且还是可省略
  • 对单个文件操作时候HEAD指向不会变
  • 将commit中指定的文件同步到暂存区中

git reset [commit-id] [file-name] 将指定commit-id中的文件替换掉暂存区的文件

4.13、revert

git revert -n [commit-id]
git revert -n HEAD

撤回版本的比较

  • reset 切换版本是会删除丢弃最新的版本的，HEAD会直接跳到指定版本，但是还是可以通过reflog找回。

  • revert 会将指定的bug版本视为bug版，会将当前版本中的bug版的代码删除，生成新的commit覆盖掉当前commit，但是commit-id是不会变的。

4.14、branch

分支可以理解为当前工作目录的一个副本，在开发过程中，一般分为主分支（main/master）和开发分支（dev/develop）：

关于git分支管理命令如下图所示：

基础命令如下：

`git branch [branch-name]` 			# 已有仓库，创建分支

`git branch` 						# 查看当前分支

`git branch -a` 					# 查看本地和远程的所有分支

`git branch -r` 					# 查看远程所有分支

`git branch -m main master` 		# 分支重命名

`git checkout master` 				# 切换分支

`git checkout -b dev` 				# 创建并切换分支

`git branch -d [branch-name]` 		# 删除一个分支

`git branch -D [branch-name]` 		# 强制删除一个没有合并的分支

`git branch --set-upstream-to=origin/[branch-name] [branch-name]` # 把本地分支和远程分支进行连接

在替换分支时，也可以使用switch命令：

`git switch -c [branch-name]` 创建新分支并切换到该分支

`git switch [branch-name]` 切换到已有分支

4.15、merge

git merge命令用于合并分支：

`git merge` # 合并本地`origin/[branch-name]`和`HEAD->[branch-name]`的代码，并同步到工作空间

`git merge [branch-name]` # 用于合并指定分支到当前分支

`git merge --quit` # 退出当前分支合并，当合并后冲突很多，要撤回合并分支就可以用这个命令

`git merge --no-ff -m [massage] [branch-name]` # 不使用Fast forward合并分支，这样会创建新的commit，所以需要massage。这样被合并的分支HEAD指向是会变的。

注：如果使用了Fast forward方式合并分支，那么删除次要分支的时候历史分支记录也会被删除，这样就无法追寻分支合拼信息了。

4.16、fork

fork命令用于快速复制别人的仓库，并实现PR。

  • fork不是Git操作，而是一个GitHub操作，是服务端的代码仓库克隆。
  • fork后会在自已的github账户创建一个新仓库， 它包含了原来的仓库(即upstream repository，上游仓库)所有内容，如分支、Tag、提交历史等。
  • 你可以对fork出的仓库自由提交，并通过PR (Pull Request)贡献回原仓库。
  • 由于fork出的新仓库是基于原仓库，但二者在后续开发中可能会大相径庭，所以被称为“分叉”

4.17、stash

git stash 命令用于隐藏当前工作的修改，如果不隐藏自己修改的半成品代码，就会发生切换到别的分支后，将然后自己的半成品代码带入其他分支，这样就发生很多不必要的麻烦。

`git stash` 					# 隐藏当前工作的修改

`git stash save message` 		# 执行存储时，添加备注，方便查找，只有git stash 也要可以的，但查找时不方便识别。

`git stash list` 				# 查看隐藏的工作信息列表

`git stash drop` 				# 删除隐藏的工作信息

`git stash pop` 				# 恢复隐藏的工作信息，同时删除隐藏的工作信息

`git stash apply [stash@{0}]` 	# 恢复指定的隐藏工作信息，但是不会删除隐藏的工作信息

4.18、cherry-pick

git cherry-pick [commit-id] 这个是复制一次commit提交，然后在当前分支上重新提交一遍；也就是将指定commit的合并到当前分支；这种适用于在其他分支上修复了bug，但是这个bug在当前分支上依然存在，所以可以复制这个commit的过程，不必重写代码。

4.19、remote

remote命令用于将本地工作区关联到远程仓库 ：

`git remote add origin url` 		# origin是标识符，表示关联哪个远程仓库

`git remote (show)` 				# 查看本地添加了哪些远程分支地址

`git remote -v` 					# 查看本地添加了哪些远程分支地址更详细信息

`git remote remove origin` 			# 删除本地指定的远程地址

`git remote rename old_name new_name` # 标识符重命名

4.20、fetch

git fetch命令用于拉取远程分支最新的commit到本地仓库的origin/[branch-name]之中。

4.21、pull

git pull 从远程仓库拉取代码到工作空间

pull 和 fetch的关系

  • git pull == git fetch + git merge

在上传代码中，出现冲突问题，如下：

4.22、tag

tag就是对某个commit进行标记，相当于起别名。

添加标签的意义在于凸显某次提交的重要性，标记一个相关的提交阶段，以备将来参考；标记发布节点，用于项目发布。

`git tag` 								# 查看所有标签

# 添加标签
`git tag [version]` 					# 给当前最新的commit打上标签，HEAD所指向的提交

`git tag [version] [commit-id]` 		# 给指定的commit-id打上标签

`git tag -a [version] -m [massage] [commit-id]` # 注释标签，给指定的commit-id打上标签并附上说明文字

# 查看标签
`git tag -l {匹配的内容*}`				 # 匹配并列出标签

`git show tag_name`						# 查看标签所有信息

# 推送标签
`git push origin tag_name1 tag_name2`	# 推送指定标签

`git push origin --tags`				# 推送所有标签

# 删除标签
`git tag -d [version]` 					# 删除本地标签

`git push origin -d tag_name`			# 删除远程仓库标签

由于标签对应着项目发布，因此在创建标签时需要在远程仓库界面添加项目发布版。

4.24、rebase

git rebase -i HEAD~2 合并前两个历史提交，会弹出vim修改信息，修改第二行的pick为s，或者为squash，squash为合并的意识，然后保存退出编辑，会打开第二个vim编辑，合并并修改commit内容，保存退出会产生一个新的commit id，这样就合并了两个commit

5.1 忽略文件的版本跟踪

gitignore-忽略文件的版本跟踪

step1—>子啊项目根目录创建.gitigbore文件

step2—>配置文件信息

  • 每项配置单独占一行
  • 每行内容可以是：文件/目录的名称、路径或者其对应的模式匹配

关于模式匹配：

# 用于注释

\表示转义

*可以匹配任意字符（0或多次）
？可以匹配任意字符（1次）
注：两者都不可以匹配/

/用于分隔目录：
	当/在开头，表示从.gitignore文件所在目录开始匹配；否则，下级将全部匹配
	当/在末尾时，只匹配目录；否则，同名目录和文件都将被匹配

原先被排除的文件，使用!模式后该文件将会重新被包含。但如果该文件的父级目录被排除，那么使用!也不会再次被包合。

[]通常用于匹配一个字符列表，如: a[mn]z 可匹配 amz 和 anz.6.通常用于匹配

**用于匹配多级目录，如 a**/b 可匹配“a/b”、"a/x/b”、"a/x/y/b"等
